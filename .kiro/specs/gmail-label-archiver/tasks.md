# Implementation Plan

- [x] 1. Foundation: clasp プロジェクトと共有定数のセットアップ
- [x] 1.1 clasp プロジェクト構成と Apps Script マニフェストを用意する
  - `.clasp.json`（`rootDir: "src"`、scriptId はプレースホルダ）と `src/` ディレクトリを作成する
  - `src/appsscript.json` に `runtimeVersion: V8`、タイムゾーン、必要 OAuth スコープのみ（`https://mail.google.com/`、`spreadsheets.readonly`、`script.scriptapp`）を宣言する
  - 外部ライブラリ・課金サービスの依存を一切含めない（組み込みサービスのみ）
  - 完了状態: `clasp push` が成功し、Apps Script 側にマニフェストとスコープが反映される
  - _Requirements: 5.3_

- [x] 1.2 共有定数モジュールを定義する
  - Script Property キー（`CONFIG_SPREADSHEET_ID` / `CONFIG_SHEET_NAME`）、既定シート名、ヘッダ列名（`ラベル名` / `保持日数`）を定義する
  - 実行全体の上限 `MAX_THREADS_PER_RUN`（既定 300）、`ARCHIVE_BATCH_SIZE`（100）、`ARCHIVE_SEARCH_LIMIT`（500）、`TRIGGER_HANDLER`、`TRIGGER_HOUR`（既定 3）を定義する
  - 完了状態: すべての定数がグローバルから参照可能で、後続モジュールがハードコードを持たない
  - _Requirements: 5.1_
  - _Boundary: Constants_

- [ ] 2. ConfigService: 設定シートの読取とバリデーション
- [x] 2.1 (P) 行バリデーションの純粋ロジックを実装し単体テストする
  - 1 行分のラベル名・保持日数を検証し、有効なら検証済みルールに変換、無効なら除外する純粋関数を実装する
  - 空ラベル、非数値・負・空の保持日数を無効と判定し、前後空白を trim、保持日数は整数かつ正のみ許可する
  - 完了状態: 正常行／空ラベル行／不正日数行を網羅した単体テストが全て通過する
  - _Requirements: 1.2, 1.4, 1.5_
  - _Boundary: ConfigService_
  - _Depends: 1.2_

- [ ] 2.2 設定シートを読み取り検証済みルール配列を返す
  - Script Properties から設定スプレッドシート ID（必須）とシート名（任意・既定値）を取得する
  - 対象シートを 1 行目ヘッダ＋2 行目以降データとして読み、ヘッダ名で列を解決し、各行を 2.1 のバリデーションに通す
  - データ行 0 件・全行無効なら空配列を返す（後段で何もせず正常終了）／ID 未設定・シート不在は例外として早期に表面化させる
  - 完了状態: 有効行・無効行混在のシートから有効行のみのルール配列が返り、空シートで空配列が返る
  - _Requirements: 1.1, 1.2, 1.3, 1.4, 1.5_
  - _Boundary: ConfigService_

- [ ] 3. ArchiveService: 対象抽出・権威判定・バッチアーカイブ
- [ ] 3.1 (P) 検索クエリ生成と日時／スター判定の純粋ロジックを実装し単体テストする
  - 検索クエリ生成（`label:"<name>" in:inbox -is:starred older_than:<days>d`、ラベル名は常に二重引用符で囲む）を実装する
  - 最新メッセージ日時がしきい値より古いかの判定（経過日数が保持日数ちょうど以下は除外）と、スター付き有無の判定を実装する
  - 完了状態: 空白入りラベルの引用、しきい値境界（以下は除外／超過は対象）、スター有無を網羅した単体テストが全て通過する
  - _Requirements: 2.1, 2.2, 2.3, 2.5, 2.6_
  - _Boundary: ArchiveService_
  - _Depends: 1.2_

- [ ] 3.2 1 ルール分の抽出・判定・残予算内アーカイブを実装する
  - 残予算の範囲で Gmail 検索（`min(remaining, ARCHIVE_SEARCH_LIMIT)`）し、候補をコード側で最新日時超過かつ非スターに絞り込む
  - 確定対象を渡された残予算まで採用し、`ARCHIVE_BATCH_SIZE`（100）単位で受信トレイから外す（標準アーカイブのみ。削除・転送・既読変更・他ラベル変更を行わない）
  - ラベル不在は検索 0 件として安全に空動作、対象 0 件は正常終了、各ルールの処理件数を標準ログへ出力する
  - 完了状態: テスト用ラベル＋古いメールで実行後に対象が受信トレイから外れ、スター付き・新しいメール・INBOX 以外のラベルが保持される
  - _Requirements: 2.1, 2.2, 2.3, 2.4, 2.5, 2.6, 3.1, 3.2, 3.3, 3.4, 5.1, 5.4_
  - _Boundary: ArchiveService_
  - _Depends: 3.1_

- [ ] 4. Main: エントリのオーケストレーションとトリガー管理
- [ ] 4.1 トリガー／手動共通のアーカイブエントリを実装する
  - 設定読取後、残予算を `MAX_THREADS_PER_RUN` で初期化し、各ルールを残予算付きで処理して処理件数を減算、残予算枯渇で以降のルールをスキップする
  - ルール単位 try/catch でエラーを記録して次ルールへ継続し、有効ルール 0 件なら何もせず正常終了する
  - 実行サマリ（ルール別件数・スキップ・エラー）を標準ログに出力する。トリガーからも手動からも同一処理とする
  - 完了状態: 複数ルール実行で全体合計アーカイブ件数が `MAX_THREADS_PER_RUN` を超えず、1 ルールの失敗が他ルールを止めない
  - _Requirements: 1.3, 4.3, 5.1, 5.2, 5.4_
  - _Boundary: Main_
  - _Depends: 2.2, 3.2_

- [ ] 4.2 日次トリガーの冪等なセットアップ／削除を実装する
  - 既存の同一ハンドラのトリガーを全削除してから日次トリガーを 1 件作成するセットアップ関数を実装する
  - 同一ハンドラの全トリガーを削除する運用補助関数を実装する
  - 完了状態: セットアップを 2 回実行しても対象ハンドラのトリガーがちょうど 1 件のみ存在する
  - _Requirements: 4.1, 4.2_
  - _Boundary: Main_
  - _Depends: 4.1_

- [ ] 5. Validation: 手動統合シナリオ検証
- [ ] 5.1 実 Gmail／Spreadsheet を用いた統合シナリオを検証する
  - 設定シート読取（有効行のみ返却・空シートで空動作）、アーカイブ実行（対象が外れ、スター付き・新しいメールが残る）を確認する
  - 非破壊確認（削除・既読化・他ラベル変更が起きていない）、トリガー冪等性、設定 → 手動実行 → 翌日トリガーで上限超過分が処理される持ち越しを確認する
  - 完了状態: 上記シナリオの実行ログとメール状態が期待どおりであることを確認できる
  - _Requirements: 1.1, 1.3, 2.1, 2.2, 2.4, 2.6, 3.1, 3.2, 3.3, 3.4, 4.2, 4.3, 5.1_
  - _Depends: 4.2_
