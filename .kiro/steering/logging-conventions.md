# ログ出力規約

実行ログ（GAS の Cloud Logging / 実行記録）の書き方を統一するための規約。新規コードはこのパターンに従う。網羅的なメッセージ一覧ではなく、判断基準とパターンを示す。

## 方針

- 運用者（=ユーザー本人）が GAS の実行ログを見て「何が起きたか」を日本語で即把握できることを最優先する。
- ログは絵文字 + タグ + 日本語の 1 行で統一し、grep / 目視の両方で拾いやすくする。
- 1 つの出来事につき 1 行・1 回だけ出力する（二重出力を作らない）。

## 出力先は `console.log` に統一（`Logger.log` 禁止）

すべてのログは `console.log` から出力する。`Logger.log` は使わない。

- **理由**: GAS V8 ランタイムでは `console.log` も `Logger.log` も同じ Cloud Logging（実行ログ）へ書き込まれる。両方を呼ぶと同一行が二重出力される。出力先を `console.log` 一本に絞ることで二重出力を防ぐ。
- Node のテスト環境でも `console.log` は 1 回だけ出力されるため、GAS / Node 双方で挙動が揃う。

直接 `console.log` を散在させず、専用のログ関数（`logLine` / `logSummary` など）を経由する。これらは `typeof console !== 'undefined' && console.log` をガードしてから出力する。

```js
function logLine(line) {
  if (typeof console !== 'undefined' && console.log) {
    console.log(line);
  }
}
```

- **理由**: Node のテストや将来の実行環境で `console` が無い場合でも安全に動かすためのガード。ログ整形を 1 箇所に集約し、タグ規約の逸脱を防ぐ。

## メッセージ形式: 絵文字 + タグ + 日本語

各ログ行は次の構造にする。

```
<絵文字> [タグ] <日本語の本文（必要なら件数・ラベル名などの数値情報）>
```

### 現行タグ（コードで使用中）

| 絵文字 + タグ | 用途 | 出力箇所の例 |
|---|---|---|
| `✅ [アーカイブ完了]` | ルール単位のアーカイブ結果（件数報告） | `logSummary`（Archiver.js） |
| `📊 [サマリ]` | 実行全体のサマリ（処理ルール数・エラー数・残予算） | `archiveLabeledThreads`（Main.js） |
| `⚠️ [エラー]` | 設定読取失敗・ルール単位の処理失敗 | `archiveLabeledThreads`（Main.js） |
| `ℹ️ [スキップ]` | 対象ルール 0 件などで処理を行わなかった旨 | `archiveLabeledThreads`（Main.js） |

実例:

```js
// 結果報告（✅）
'✅ [アーカイブ完了] ラベル「' + labelName + '」: アーカイブ ' + archivedCount + '件 / 候補 ' + candidateCount + '件';

// 全体サマリ（📊）— 複数行は "\n   └ " で字下げして続ける
'📊 [サマリ] ルール ' + rules.length + '件中 処理成功 ' + processedCount + '件 / エラー ' + errorCount + '件 / 残り処理可能 ' + remaining + '件\n   └ ルール別: [' + perRule.join(', ') + ']';

// エラー（⚠️）— 例外メッセージは errorMessage(err) で安全に文字列化して付加
'⚠️ [エラー] ラベル「' + rule.labelName + '」の処理中にエラーが発生しました: ' + errorMessage(err);

// スキップ（ℹ️）
'ℹ️ [スキップ] 対象のルールが無いため、処理を実行しませんでした。';
```

### 形式ルール

- 絵文字と `[タグ]` は半角スペースで区切り、`[タグ]` の直後も半角スペース 1 つ。
- 本文は日本語。ラベル名は `「...」` で囲む。件数は `N件` の形で末尾に数値を添える。
- 例外メッセージはそのまま連結せず `errorMessage(err)` を通す（`Error` 以外も安全に文字列化するため）。
- 新しいタグを足す場合も「絵文字 + `[日本語タグ]`」の形を守り、上の語彙（完了 / サマリ / エラー / スキップ）と整合させる。意味が重複するタグを増やさない。

## エラーハンドリングとログの対応

ログの粒度は本プロジェクトのエラー戦略（tech.md「フェイルファスト vs 継続」）に対応させる。

- **構成不備**（設定読取失敗など）: `⚠️ [エラー]` を 1 行出力し、再 throw せず正常終了する。
- **ルール単位の失敗**: `⚠️ [エラー]` を出力して次ルールへ継続し、最後に `📊 [サマリ]` でエラー件数を必ず合算報告する。
- **正常時**: ルールごとに `✅ [アーカイブ完了]`、実行末尾に `📊 [サマリ]` を 1 回。

- **理由**: 個別のエラー行（⚠️）だけでなく実行末尾のサマリ（📊）でも件数が突き合わせられるため、ログを上から読まなくても「全体でいくつ失敗したか」が分かる。
