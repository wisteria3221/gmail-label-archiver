# 手動統合検証ランブック（タスク 5.1）

実 Gmail / Spreadsheet を用いた統合シナリオ検証の手順。自動化された純粋ロジックの単体テスト（`npm test`）でカバーできない、GAS サービス連携部分を実環境で確認する。

## 前提（利用者が事前に用意）

- Google アカウント（個人 Gmail）
- clasp をインストールし認証済み: `npm i -g @google/clasp && clasp login`
- 設定用スプレッドシートを 1 つ用意し、1 行目ヘッダ `ラベル名 | 保持日数`、2 行目以降に検証用ルールを記入
- Gmail 側に検証用ラベルを作成・付与（古いメール／新しいメール／スター付きを各 1 通以上）

## サンプル設定シート

[docs/sample-settings.csv](sample-settings.csv) を Google スプレッドシートにインポート（ファイル → インポート → アップロード → 「現在のシートを置換」）すると、検証用の行が一括で入る。各行の意図は次のとおり。

| 行 | ラベル名 | 保持日数 | 検証シナリオ | 期待挙動 |
|----|---------|---------|------------|---------|
| 1 | （ヘッダ） | （ヘッダ） | — | 1 行目はヘッダとして読み飛ばされる |
| 2 | `アーカイブ/ニュースレター` | `30` | S2, S3 | 有効行。30 日超過の非スターを受信トレイから外す。新しい/スター付き/他ラベルは保持 |
| 3 | `アーカイブ/通知` | `7` | S1, S2 | 有効行。7 日しきい値で動作 |
| 4 | `アーカイブ/大量` | `1` | S5 | 有効行。保持 1 日。`MAX_THREADS_PER_RUN` 超の古いメールを入れて持ち越しを確認 |
| 5 | `存在しないラベル` | `14` | S2 (2.4) | Gmail に無いラベル。検索 0 件でスキップ、他行は継続 |
| 6 | （空） | `5` | S1 (1.5) | 空ラベル → 行スキップ |
| 7 | `アーカイブ/不正日数` | `abc` | S1 (1.4) | 非数値 → 行スキップ |
| 8 | `アーカイブ/負の日数` | `-1` | S1 (1.4) | 負値 → 行スキップ |
| 9 | `アーカイブ/ゼロ日数` | `0` | S1 (1.4) | 0 → 行スキップ |

> 事前に Gmail 側で `アーカイブ/ニュースレター`・`アーカイブ/通知`・`アーカイブ/大量` のラベルを作成し、各ラベルに「保持日数を超えた古いメール」「保持日数内の新しいメール」「古いがスター付きのメール」を付与しておく。行 5〜9 は意図的な不正/不在行なので Gmail 側の準備は不要。

## セットアップ

1. `.clasp.json` の `scriptId` を自分の Apps Script プロジェクト ID に置き換える。
2. `clasp push` で `src/` を Apps Script へ反映。
3. Apps Script エディタ → プロジェクトの設定 → スクリプト プロパティに以下を登録:
   - `CONFIG_SPREADSHEET_ID` = 設定スプレッドシートの ID（必須）
   - `CONFIG_SHEET_NAME` = シート名（省略時は `settings`）
4. 初回実行時に OAuth 同意（Gmail / Spreadsheet / トリガー権限）を承認。

## 検証シナリオ

各シナリオ実行後、Apps Script の「実行ログ」と Gmail の実際の状態を確認する。

### S1. 設定シート読取（要件 1.1, 1.3, 1.4, 1.5）
- 有効行・無効行（空ラベル／保持日数 `abc`・`-1`・`0`）を混在させる。
- `archiveLabeledThreads` を手動実行 → ログで有効行のみが処理対象になり、無効行がスキップされていること。
- データ行を空（ヘッダのみ）にして実行 → 何も処理せず正常終了すること（1.3）。
- `CONFIG_SPREADSHEET_ID` を一時的に未設定にして実行 → ログに構成不備エラーが記録され、処理が安全に終了すること（フェイルファスト）。

### S2. アーカイブ実行（要件 2.1, 2.2, 2.4, 2.5, 2.6, 3.1, 3.4）
- 対象ラベルに「保持日数超過の古いメール」「保持日数内の新しいメール」「古いがスター付きのメール」を用意。
- 手動実行後:
  - 古い非スターのスレッドが受信トレイから外れている（アーカイブ済み）。
  - 新しいスレッドは受信トレイに残っている（2.5）。
  - スター付きスレッドは受信トレイに残っている（2.6）。
- 存在しないラベル名を設定行に追加して実行 → その行はスキップされ他行は継続（2.4）。

### S3. 非破壊確認（要件 3.2, 3.3）
- S2 実行後、対象スレッドが:
  - 削除されていない／ゴミ箱に入っていない（3.3）。
  - 既読状態が変わっていない（3.3）。
  - INBOX 以外の既存ラベルが保持されている（3.2）。

### S4. トリガー冪等性（要件 4.1, 4.2）
- `setupDailyTrigger` を実行 → トリガー一覧に `archiveLabeledThreads` の日次トリガーが 1 件。
- もう一度 `setupDailyTrigger` を実行 → 依然として 1 件のみ（重複が増えない）。

### S5. 手動＝トリガー同一処理／持ち越し（要件 4.3, 5.1）
- 対象スレッドを `MAX_THREADS_PER_RUN`（300）より多く用意（または定数を一時的に小さくして検証）。
- 手動実行 → 上限件数までアーカイブされ、超過分は受信トレイに残る。
- 翌日のトリガー（または再度手動実行）→ 残っていた超過分が処理される（冪等な持ち越し）。

## 合否

すべてのシナリオで実行ログとメール状態が期待どおりであれば、タスク 5.1 を完了とする。
